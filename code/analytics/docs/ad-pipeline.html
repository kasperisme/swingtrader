Stage detail

1

News → Trend

nis-trend-radar

Reads: news_trends_tag_daily_v (themes/day) + news_trends_ticker_daily_v (ticker mentions/day), enriched by news_impact_heads (cluster='STORY_KEY_POINTS' → why it's trending), ticker_sentiment_heads_v (topic-scoped sentiment) and market_screenings (CTA matching).

Logic: current vs prior 7-day windows → heat = current × (1 + 2·growth) (tilts toward acceleration) → excludes ~40 generic process tags (earnings, guidance, lawsuit…) → volume floor (topic ≥ 8 articles).

The feedback loops

Loop A · performance → creative

Winning levers bias the next genome

design --json → next ad's design block. Closed by the ad_id ↔ design.json join. Automated read, manual apply — vary one lever per variant so attribution stays clean.

Loop B · trend → product

Story with no screen → new screen

When the winning topic has no matching screener, trend_brief sets needs_new_screening + a ready-to-build suggested_screening spec → you create it → it becomes a new lead-magnet destination for future ads.

Loop C · fatigue → refresh

Worn creative → fresh variant

design's frequency / ⚠fatigue signal tells you when a creative is spent and needs a fresh variant appended to the winning ad set (preserves learning) rather than a brand-new ad set.

The through-key

One id, four stages

The whole loop hangs on ad_id in launch_manifest.json linking Meta spend back to the exact creative genome that produced it. No id, no feedback.

Manual seams (not automated)

  1. Director picks the story & authors ad.json. The trend brief is ready-to-use, but the creative call is human.
  2. You flip ads Active in Ads Manager. Nothing in the pipeline can spend — everything is created PAUSED.
  3. Step 0 is a read, not an auto-apply. The next genome is chosen from the leaderboard by you, not written automatically.
